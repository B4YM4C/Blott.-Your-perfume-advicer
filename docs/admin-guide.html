01

Quick start (30 seconds)

The app is committed under blott-app/. Local mode uses an in-memory mock DB seeded from JSON, so no MySQL/Postgres required to play with the admin UI.

# 1. install once
cd blott-app
npm install

# 2. (optional) copy env template — already includes a .env.local default
cp .env.example .env.local       # skip if .env.local already exists

# 3. start dev server
npm run dev

# → ready on http://localhost:3000
# → admin   on http://localhost:3000/admin
Beta convention

No login wall yet. Anyone with the URL can edit. Lock it down before staging — see section 08.

04

Manage questions

Each question has: order · title (TH) · subtitle (EN) · optional image URL · 2–6 choices (codes A → F). When you delete a choice, the remaining ones get re-coded automatically.

1

Add a question

Open /admin/questions → click + Add question top-right. Fill in title + at least 2 choices → Save question.

2

Edit a question

Click Edit → on any row. Change anything; press Save question.

3

Add a choice

Inside the editor → + Add choice. Letter codes A–F are assigned automatically — you don't pick them.

4

Remove a choice

Remove next to the choice. Two choices is the minimum (the button disables below that).

5

Delete a question

Delete at the bottom of the editor. In production this is a soft delete (sets deleted_at) so historical answers stay intact.

Adding question 6 doesn't break anything

Answers are stored 1-row-per-question, not as a wide table — so changing question count never requires a migration. Just remember to refresh your result mappings to cover the new code.

Result mappings

Each mapping is a row of: pattern → fragrance + house + family + notes + blurb + image. Open /admin/mappings.

Click + Add mapping for a new row. Click Remove to delete one. Important: nothing saves until you press Save all — the editor sends the entire list as a single PUT request and replaces the table atomically.

06

Pattern cheat-sheet

Exact match

User picked B, B, C, D, B

Pattern: 1B2B3C4D5B → first exact match wins.

Wildcard

Any user who picked C on Q3

Pattern: 1*2*3C4*5* → matches any combo of choices on the other questions.

Group

Formal + Mysterious

1*2*3D4B5* → catches formal-dressed mysterious types regardless of budget/age/location.

Catch-all

Nothing else matched

Pattern: default → fallback. Always have one or the user sees an error.

Matching order in lib/quizLogic.js: exact → wildcard (first match) → default.

07

Where data goes

APP_MODE = local
Default · dev / demo

In-memory Maps inside the Node process. Seeded from data/questions.json + data/mappings.json on first read.

✓ Zero setup · ✓ Edits in admin work immediately · ✗ Everything resets when you restart npm run dev.

Want changes to survive restart? Either switch to production mode, or export and overwrite the JSON files manually.

APP_MODE = production
Persistent · staging / live

SQL via mysql2 or pg. Schema in sql/schema.sql — 8 tables: users, sessions, questions, choices, answers, result_mappings, results, tracking_events, consent_log.

Both adapters expose the same async interface (see lib/db/index.js) — your routes/components don't change one bit when you swap.

08

Going to production

1

Create database + user

MySQL example below. Postgres equivalent in sql/schema.sql comments.

mysql -u root -p
CREATE DATABASE blott CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'blott'@'%' IDENTIFIED BY 'change_me';
GRANT ALL ON blott.* TO 'blott'@'%';
FLUSH PRIVILEGES;
2

Apply schema + seed

mysql -u blott -p blott < sql/schema.sql

# or via the Node helper:
APP_MODE=production node scripts/init-db.js
APP_MODE=production node scripts/seed.js
3

Switch app mode

Edit .env.local (or your deploy env):

APP_MODE=production
DB_DRIVER=mysql
DB_HOST=…
DB_USER=blott
DB_PASSWORD=…
DB_NAME=blott
ADMIN_USERNAME=…
ADMIN_PASSWORD=…
4

Build + run

npm run build
npm run start  # serves on :3000
5

Lock /admin

Beta has no auth gate. Before staging, add a Next middleware that rejects requests to /admin/** + /api/admin/** unless basic-auth credentials match ADMIN_USERNAME / ADMIN_PASSWORD from env. Or put the whole back-office behind a VPN / IP allowlist.

09

Troubleshooting

"Cannot find module '@/lib/db'"

Make sure you ran npm install inside blott-app/. The @/ alias is defined in jsconfig.json; restart npm run dev after touching it.

Edits in /admin don't persist

You're in APP_MODE=local — that's expected. The mock DB resets on every Node restart. Switch to production mode (section 08) for persistence.

Result page says "no match"

No mapping matched the user's pattern and there's no default row. Add a default mapping in /admin/mappings.

User finished quiz but result is wrong

Open the dashboard → check the Sessions count went up. Then walk through your mappings — the matcher tries exact first, then iterates wildcard rows in the order they appear, then falls back to default. If two wildcards both match, the earlier one wins. Reorder rows accordingly.

Email step is blocking testers

Beta intentionally exposes a Skip button on the email step. If you want to force email collection later, edit app/quiz/QuizClient.jsx → remove the Skip button from EmailStep.

Tracking is silent

Two gates: (1) TRACKING_ENABLED=true in env, (2) the user accepted the consent banner. Either says no → /api/tracking writes nothing.
